# Stream Continuation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Continue eligible interrupted `/v1/completions` SSE responses from their emitted text prefix using an independently bounded retry phase.

**Architecture:** Add endpoint-scoped continuation configuration to `FallbackConfig`, extract reusable provider-selection state from the existing fallback iterator, and implement protocol parsing plus body orchestration in a focused `stream_continuation` module. The request handler supplies the original request context and keeps response sanitization outside the continuation wrapper so all upstream attempts follow one downstream transformation path.

**Tech Stack:** Rust 2024, Axum bodies, Tokio timeouts, `async-stream`, Serde JSON, existing provider pool and retry primitives.

## Global Constraints

- Continuation is disabled by default and requires both `fallback.enabled` and `stream_continuation.enabled`.
- Only streaming `POST /v1/completions` requests with a string prompt, one choice, and no prompt echo are eligible.
- Never log prompt text, generated text, request bodies, or response bodies.
- Buffer at most `max_buffered_bytes` generated bytes per eligible response; the default is 1 MiB.
- Pre-response fallback and post-commit continuation have independent attempt and backoff budgets.
- A terminal `[DONE]` event or non-null `finish_reason` prevents continuation.
- Do not synthesize aggregate usage or a successful terminal marker.

- [ ] **Step 3: Implement the body wrapper and request attempts**

Before non-strict response transformation in `target_message_handler`, detect an
eligible successful SSE response and replace its body with
`wrap_completion_stream`. The wrapper must:

1. Wrap each upstream body in `SseBufferedStream`.
2. Forward complete events immediately while updating protocol state.
3. Treat premature EOF, body error, and configured idle timeout as interruption.
4. Drop the active provider guard before selecting another provider.
5. Use `SelectionState` with `stream_continuation.max_attempts`.
6. Apply provider rate/concurrency limits, model rewrite, auth headers, trace
   policy, header timeout, retryable status rules, backoff, and backoff cap.
7. Reuse the original downstream stream identity on continuation chunks.
8. Keep the inflight guard alive until the combined body is dropped.

Change `GuardedStream` to hold `Option<ConcurrencyGuard>`: non-eligible responses
retain `Some(connection_guard)` there, while eligible responses move the guard
into the continuation wrapper and pass `None` to the outer guard.
